## 验证标准

### Backend 完整 check

- `cd backend && make check` 全绿（lint + test + build）
- `golangci-lint run ./...` 返回 0 issues
- `go test -race ./...` 通过
- DB migration roundtrip：`make migrate-down; make migrate-up; make migrate-down; make migrate-up` 无错

### Frontend 完整 check

- `cd frontend && pnpm lint:all` 0 warnings
- `cd frontend && pnpm test` 所有单测通过
- `cd frontend && pnpm build` 成功
- bundle size 变化 < 50kb（如果超过需要在执行报告里说明原因）

### 端到端烟雾测试

在本地 docker-compose 启动环境，按顺序执行：

1. 重建库：`make migrate-down && make migrate-up`
2. 启动 backend：`make dev`（另一个终端）
3. 启动 frontend：`cd frontend && pnpm dev`（另一个终端）
4. 注册新用户 A
5. Onboarding 选"跳过 AI"
6. 触发分析（需要先添加一个测试 holding）
7. 验证卡片显示"Rules"角标
8. 进 settings → 配置 LLM（用 test key，或者提前 export env 让 stub provider 响应）
9. Probe 成功 → 保存成功
10. Dashboard 顶部出现 banner
11. 点击"重新分析所有持仓"
12. 分析完成后卡片显示"AI"角标，banner 消失
13. 模拟 key 失效（手工改 DB 让 health_status=failing 或者替换 mock）
14. 再次分析 → 如果勾了 fallback → 卡片仍是"AI"（走 system_default），如果没勾 → 卡片变"Rules"
15. 清理：删除测试用户的数据

### 指标验证

- 访问 `/metrics` endpoint（如果后端有暴露）
- 检查 `llm_request_total`、`llm_fallback_total`、`decision_cards_by_source` 指标是否按预期增长
- 如果后端还没暴露 /metrics，这一步记录到"未来工作"

### 零 AI 痕迹扫描

```
git diff main..HEAD | grep -iE "(generated by|ai-assist|copilot|anthropic-authored|claude-authored)"
```

应该返回空输出。如果有匹配，检查并修复。

**允许的字样**：Claude API / OpenAI API（作为产品名）、"claude" 包路径名（如 `internal/llm/claude/`）。禁止的字样：任何暗示 "本代码由 AI 生成" 的句子。

### 执行报告

在 `docs/reports/llm-degraded-contract-execution-report.md` 里完整记录：

1. **执行上下文**：worktree 路径、分支名、基于的 main SHA、开始/结束时间
2. **全局规则**：commit 规则、lint 规则、零 AI 痕迹
3. **每个 step 的执行结果**：commit SHA 列表、关键决策、偏差说明、验收结果
4. **已修复的问题**：每个修复的问题的描述和验证
5. **已记录但未修复的观察项**：原因、后续处理建议
6. **无法决策项**：等待用户验收时决策
7. **Review 结果**：spec compliance pass / code quality pass
8. **总结**：可合并 yes/no、风险评估

### 用户验收

执行报告完成后，主会话把报告摘要呈现给用户，等待验收。用户确认后才触发合并到 main。

## 合并顺序

按依赖顺序合并到 main：

1. `chore/golangci-lint-v2-and-cleanup` → main（先合并，清理 baseline）
2. `docs/llm-degraded-contract` → main（再合并，落盘文档）
3. `feat/llm-degraded-contract` → main（最后合并，代码实现）

每次 merge 使用 fast-forward（如果可能）或 non-ff merge commit，保留分支历史。
