# 百万 · Richman — 产品需求文档（PRD）

> 版本：2.0 · 2026年4月 · 完整自包含版本

---

## 1. 产品概述

### 1.1 产品名称

百万 · Richman

### 1.2 产品定位

AI 驱动的个人投研决策助手。

### 1.3 核心价值

把专业基金经理的思维方式装进普通人口袋——不做信息聚合，不做交易执行，专注两件事：

1. **告诉你市场在发生什么、为什么**（免费开放，吸引用户）
2. **基于你的持仓，告诉你该怎么做**（持仓用户专属价值）

### 1.4 与通用 AI 助手的差异化

用户可以直接问 ChatGPT/Claude"我该不该买黄金"并得到一个像样的回答。Richman 的差异化在于五个通用 AI 做不到的事：

1. **量化一致性**：通用 AI 每次回答可能不同。Richman 基于同一套 FRED 数据 + 百分位排名 + 权重模型，评分可复现、可追溯
2. **时间序列追踪**：通用 AI 没有记忆。Richman 有 90 天得分趋势线、变化归因、模型版本历史——用户能看到"上周 68 分，这周 75 分，主要因为央行购金"
3. **持仓个性化**：通用 AI 不知道你持有多少。Richman 基于真实持仓生成具体到手数的执行计划、止损位、集中度检查
4. **主动推送**：通用 AI 需要你去问。Richman 每天推送评分变化和市场快讯，用户不会错过关键信号
5. **制度化风控**：通用 AI 可能给出过度乐观的建议。Richman 有集中度分级检查、止损规则、冲突检测、风险偏好适配等系统化风控机制

有钱、有逻辑、但缺少投资经验的城市白领。

用户特征：
- 月收入较高，持有一定规模投资资产
- 逻辑思维强但不懂专业投研方法论
- 希望得到有依据的决策建议而非模糊推荐

### 1.5 商业化模式

- MVP 阶段：邀请制，需邀请码注册
- 后续阶段：免费层（行情分析）+ 付费订阅（个性化执行计划）

### 1.6 国际化

- MVP：中文 + 英文
- 后续：按需扩展更多语言

### 1.7 核心原则

- **框架完整，灯只亮一盏**：产品框架覆盖全部标的类别，但 MVP 只激活黄金标的，其他置灰预留
- **平台预计算，用户零成本访问**：所有标的级分析由平台定时计算并缓存，用户访问不触发实时计算
- **免费给判断，持仓给行动**：分析评分、维度解读、风险提示免费开放；个性化执行计划需录入持仓

## 3. 分析引擎

### 3.1 架构设计

采用**量化底座 + LLM 智能层 + 综合输出**的三层架构：

**Layer 1 - 量化底座（确定性）**：由独立的 Python 量化服务 **richson** 承担。richson 拥有全部金融数据的获取、处理和计算职责——FRED 宏观指标、Yahoo Finance / AKShare 价格数据、Polymarket 事件概率、CFTC COT 持仓数据的拉取，频率对齐、百分位排名标准化、技术指标计算、维度评分计算均在 richson 内完成。Go 后端（richman）不直接调用任何金融数据 API，只向 richson 请求计算结果。使用 Python 是因为其在量化金融领域的工具生态（statsmodels、pandas-ta、riskfolio-lib 等）远超 Go 生态，且产品路线中规划的多因子回归、体制检测、回测验证等高级功能在 Go 中无成熟替代方案。具体的服务架构和通信方式在 TRD 中定义。

**Layer 2 - LLM 智能层（信息检索 + 分析研判）**：LLM 不仅是文本生成工具，更是**信息检索与分析引擎**。LLM 利用联网搜索能力，按照产品预定义的检索策略主动获取实时信息（央行购金新闻、美联储官员讲话、地缘政治事件、机构研报等），分析提炼后对量化底座的维度得分进行有限度调整（±15 分上限），并提供调整理由。这一层保持信息的新鲜度和信息扩展的弹性——无需预先硬编码所有数据源，LLM 可以根据检索策略动态发现和整合新信息。

**Layer 3 - 综合输出**：合成量化基础分和 LLM 调整后的最终评分，生成综合解读文本、风险因子、事件日历，输出决策卡片。

**降级原则**：LLM 不可用时，Layer 2 整层跳过——量化底座分数直接作为最终评分，解读文本由模板生成。系统始终可用，LLM 是增强而非依赖。

**LLM 输出验证规则**：
- **来源可追溯**：LLM 每次调整必须附带信息来源的 URL 或明确出处。无法提供来源的调整不生效
- **双信源交叉验证**：影响程度为"重大"的判断需要 ≥ 2 个独立信源。单一信源最高触发"中等"级别调整（±8 分上限）
- **异常检测**：当 LLM 调整导致某维度单日变化 > 10 分时，标注"待验证"，用户界面显示"⚠ 此调整基于最新信息，尚待多源验证"

### 3.2 标的分析模型：每标的可定义 N 维

每个标的类型注册自己的 AssetAnalyzer，定义独立的维度集合、权重、数据源和评分逻辑。MVP 阶段只实现黄金模型，其他标的模型后续逐个开发。

### 3.3 黄金四维分析模型

#### 3.3.1 四维定义

| 维度 | 名称 | 默认权重 | 驱动逻辑 |
|---|---|---|---|
| D1 | 宏观利率 | 30% | 实际利率（TIPS）是黄金的机会成本。利率下降 → 看涨黄金 |
| D2 | 美元流动性 | 25% | 黄金以美元计价。DXY 走弱 + 流动性扩张 → 看涨黄金 |
| D3 | 结构性需求 | 25% | 央行购金、ETF 资金流等价格不敏感的需求创造结构性底部 |
| D4 | 技术位置 | 20% | 均线系统、动量指标、跨资产比价确认或否定宏观论点 |

权重可配置，存储于 `dimension_definitions` 表，后台可调不需要发版。微调范围 ±10%，四维之和始终 100%。

#### 3.3.2 归一化规则：双窗口混合百分位

所有子指标使用**双窗口混合百分位**进行归一化，解决单一长窗口混合不同市场体制的问题：

```
最终百分位 = 0.7 × 短期百分位（1年） + 0.3 × 长期百分位（5年）
```

- **短期窗口（1 年）**：反映"当前市场环境内的相对位置"，权重 70%
- **长期窗口（5 年）**：反映"更广历史背景下的绝对位置"，权重 30%
- 对于"越低越利好黄金"的指标（TIPS 收益率、DXY 等），取 `100 − 混合百分位`
- 最终：0 = 最不利于黄金，100 = 最有利于黄金
- 70/30 混合比例偏重短期，因为投资决策更依赖当前环境下的相对位置

#### 3.3.3 D1 宏观利率（总权重 30%）

| 子指标 | 数据源 | FRED 代码 | 维度内权重 | 评分逻辑 |
|---|---|---|---|---|
| 10Y TIPS 收益率 | FRED | `DFII10` | 35% | 5 年百分位取反（越低越利好） |
| 实际联邦基金利率 | FRED | `DFF` − `PCEPILFE` | 20% | 负实际利率 → 高分 |
| 收益率曲线利差 | FRED | `T10Y2Y` | 15% | 倒挂（< 0）→ 衰退恐惧 → 利好 |
| 10Y 通胀预期 | FRED | `T10YIE` | 15% | 上升 → 利好 |
| 5Y 通胀预期 | FRED | `T5YIE` | 10% | 上升 → 利好（与 10Y 互补） |
| Polymarket 降息概率 | Polymarket API | — | 5% | 降息概率 > 60% → 利好 |

**D1 LLM 智能层**：检索方向为美联储官员讲话、FOMC 会议纪要解读、利率预期变化。调整上限 ±15 分。典型调整场景："美联储主席鲍威尔最新讲话释放鸽派信号，市场降息预期升温"→ D1 量化基础分 +8~12。

#### 3.3.4 D2 美元流动性（总权重 25%）

WALCL、WTREGEN、RRPONTSYD 先合成为"净流动性"复合指标，不单独评分，避免双重计数。

| 子指标 | 数据源 | FRED 代码 | 维度内权重 | 评分逻辑 |
|---|---|---|---|---|
| 美元广义指数 (DXY) | FRED | `DTWEXBGS` | 35% | 5 年百分位取反（下降 = 利好） |
| 净流动性 | FRED | `WALCL` − `WTREGEN` − `RRPONTSYD` | 40% | 5 年百分位（上升 = 利好） |
| M2 增速 | FRED | `M2SL` / `WM2NS` | 25% | 增速加速 → 利好 |

净流动性频率对齐：WALCL（周度）和 WTREGEN（周度）前向填充到日度后与 RRPONTSYD（日度）相减。

**D2 LLM 智能层**：检索方向为美元走势分析、全球流动性事件、QT/QE 预期变化。调整上限 ±15 分。典型调整场景："日本央行意外加息推动日元走强，美元指数承压"→ D2 量化基础分 +5~10。

#### 3.3.5 D3 结构性需求（总权重 25%）

| 子指标 | 数据源 | 维度内权重 | 评分逻辑 |
|---|---|---|---|
| 央行净购买量（季度年化） | 世界黄金协会季报 | 40% | 年化 > 800 吨强烈看涨，相对 2010-2021 均值 473t 做百分位 |
| ETF 持仓变化 | yfinance (GLD/IAU) | 25% | 周度净流入趋势，4 周滚动均值的百分位 |
| COMEX COT 净多头 | CFTC 周报 | 15% | 管理资金净多头的 4 周变化方向 |
| AISC 利润率 | 世界黄金协会 | 10% | (金价 − AISC) / AISC 的百分位 |
| Polymarket 地缘事件 | Polymarket API | 10% | 地缘冲突概率上升 → 利好 |

央行购金数据为季度更新。两次季报之间，央行购金子指标得分保持不变（使用最近一期季度年化值），由 ETF 和 COT 的周度变化来调节 D3 整体得分的时效性。

**D3 LLM 智能层**：检索方向为央行购金新闻、ETF 资金流报道、地缘政治事件、黄金实物需求动态。调整上限 ±15 分。这是 LLM 价值最大的维度——央行购金季报的空窗期内，LLM 可从路透社、彭博等新闻源获取月度/周度购金动态，显著提升 D3 的时效性。典型调整场景："路透社报道中国央行 3 月增持黄金 12 吨，连续第 5 个月增持"→ D3 量化基础分 +10~15。

#### 3.3.6 D4 技术位置（总权重 20%）

| 子指标 | 工具 | 维度内权重 | 评分逻辑 |
|---|---|---|---|
| 价格 vs 200 日均线 | SMA(200) | 25% | 上方 → 75-100 分；下方 → 0-25 分 |
| 50/200 均线交叉 | SMA(50), SMA(200) | 20% | 金叉 → 80+ 分；死叉 → 20- 分 |
| 跨资产比价 | 金银比/金铜比/金油比 | 20% | 金银比 > 80 → 避险主导利好 |
| RSI(14) | 14 日相对强弱 | 15% | < 30 超卖看涨；> 70 警惕；30-70 线性映射 |
| 唐奇安通道突破 | 20 日入场 / 10 日出场 | 10% | 突破上轨 → 高分 |
| ATR(14) 信号调制 | 14 日平均真实范围 | 10% | 特殊角色，见下文 |

**ATR 的信号调制角色**：ATR(14) 用于检测波动率体制。高 ATR（> 5 年 P75）= 趋势市场，加重动量信号权重（RSI +5%、均线交叉 +5%、唐奇安 +5%，比价 -10%，ATR -5%）。低 ATR（< 5 年 P25）= 震荡市场，加重均值回归信号（比价 +5%、RSI +5%，均线交叉 -5%、唐奇安 -5%）。

**D4 不使用 LLM 智能层**：技术位置维度是纯数学计算，LLM 在此无信息优势，介入只增加噪音。D4 得分完全由量化底座决定。

### 3.4 综合评分机制

```
综合得分 = D1权重 × D1得分 + D2权重 × D2得分 + D3权重 × D3得分 + D4权重 × D4得分
```

各维度得分 = 子指标加权平均（量化基础分）+ LLM 调整（上限见下文，D4 除外）。综合得分 0-100，映射为五级信号：

| 综合得分 | 信号 | 方向标签 |
|---|---|---|
| 80-100 | 强烈看涨 | 🟢 |
| 60-79 | 中度看涨 | 🟢 |
| 40-59 | 中性 | 🟡 |
| 20-39 | 中度看空 | 🔴 |
| 0-19 | 强烈看空 | 🔴 |

**阈值校准说明**：以上等间隔阈值为初始值。上线运行 90 天后，根据综合评分的实际分布调整阈值，目标是使五个信号级别的触发频率大致为 5%/20%/50%/20%/5%，确保"强烈看涨"和"强烈看空"信号的稀缺性和决策含金量。

**LLM 调整机制**：LLM 智能层不直接输出数字调整，而是输出**结构化的定性判断**——事件描述 + 影响方向（利多/利空/中性）+ 影响程度（重大/中等/轻微）+ 置信度（高/中/低）。然后由产品定义的映射规则将定性判断转换为数字调整：

| 影响程度 | 置信度 | 利多调整 | 利空调整 |
|---|---|---|---|
| 重大 | 高 | +12~15 | -12~15 |
| 重大 | 中 | +8~11 | -8~11 |
| 中等 | 高 | +6~8 | -6~8 |
| 中等 | 中 | +3~5 | -3~5 |
| 轻微 | 任意 | +1~2 | -1~2 |

调整上限 ±15 分。LLM 只负责语义分析（它擅长的），数字由确定性规则生成（可审计、可复现）。

**评分附加信息**：
- **时间维度**：所有评分标注"中期展望（1-3 个月）"
- **评分区间条**：评分不仅展示点估计值（72），还展示一个色带区间（如 65-79），反映模型的内在不确定性。区间宽度 = 信心度的反函数（信心度越高，区间越窄）。这避免用户对个位数变化过度反应
- **历史分位**：评分旁显示自然语言分位标签（而非统计术语"PXX"）：

  | 分位区间 | 标签展示 | tooltip |
  |---|---|---|
  | P90+ | 近一年偏高 | "当前评分高于过去一年 90% 的时间" |
  | P75-89 | 近一年中高 | "当前评分高于过去一年 75% 的时间" |
  | P25-74 | 近一年中位 | "当前评分处于过去一年中间水平" |
  | P10-24 | 近一年中低 | "当前评分低于过去一年 75% 的时间" |
  | P10以下 | 近一年偏低 | "当前评分低于过去一年 90% 的时间" |
- **维度冲突检测（两层）**：
  - **强冲突**（红色警告）：任意两个维度，一个 ≥ 70 且另一个 ≤ 30
  - **弱冲突**（黄色提示）：最高与最低维度得分差距 > 40，但不满足强冲突条件
- **今日变化摘要**：当评分较上一次更新变化 ≥ 5 分时，展示变化归因——"评分 68→75（+7）：结构性需求 +5（LLM 检索到央行购金报道）、美元流动性 +3（DXY 周跌 0.8%）、技术位置 -1（RSI 走弱）"
- **重大变化说明**：当评分单次变化超过 20 分（跨越一个信号级别）时，LLM 额外生成一段复盘文本，包含：上期判断的核心依据、哪些假设被证伪、当前应如何调整。这是专业分析师在判断出错时的标准操作——复盘、归因、调整，显著提升用户信任

### 3.5 信心度计算

- 四维方向完全一致（均 ≥ 50 或均 < 50）：80-100%
- 三维一致、一维偏离：60-80%
- 两两对立：40-60%
- 三维以上方向各异：20-40%
- 数据不完整扣减：FRED 延迟 > 3 天扣 15%，Polymarket 不可用扣 5%，LLM 失败扣 10%

### 3.6 数据更新策略

| 模式 | 频率 | 触发方式 | 说明 |
|---|---|---|---|
| 日常更新 | 每日 | 美股收盘后定时任务 | 全部数据拉取 + 评分 + LLM 解读 |
| 事件触发 | 即时 | 重大事件检测 | Polymarket 关键合约概率 24h 内变化 > 20pp |
| 用户触发 | 按需 | 持仓用户手动触发 | 仅持仓分析场景 |

### 3.7 数据降级策略

| 异常场景 | 降级策略 | 对评分的影响 |
|---|---|---|
| FRED 数据延迟 ≤ 3 天 | 使用最后已知值 | 无影响，标注"数据截至 X 日" |
| FRED 数据延迟 > 3 天 | 受影响维度得分冻结 | 信心度扣 15% |
| Yahoo 价格延迟 | 使用最后收盘价 | D4 标注"数据延迟" |
| Polymarket 不可用 | Polymarket 子信号权重归零，其余重新归一化 | 信心度扣 5% |
| LLM 不可用 | 综合解读用模板生成 | 不影响评分，标注"模板生成" |
| 世界黄金协会季报未更新 | 央行购金子指标保持上期值 | 无扣减（设计预期内） |

### 3.8 模型验证要求

MVP 上线前，黄金四维模型必须通过最低限度的历史回算验证：

- **验证范围**：2020-2025 年（覆盖 COVID 冲击、加息周期、降息周期、央行购金潮四个体制）
- **验证指标**：综合评分与后续 1-3 个月金价走势的方向一致率（评分 > 60 时金价上涨，评分 < 40 时金价下跌）
- **最低标准**：方向一致率 > 60%
- **执行方**：richson 在开发阶段完成回算，结果记录在案
- **不面向用户展示**：验证结果用于内部模型校准，不作为产品功能

此验证不是完整回测（P3 功能），而是确保模型的基本合理性——一个方向一致率低于 60% 的模型不应该发布给用户。

### 3.9 模型版本管理

分析模型的权重和参数可在线调整，每次调整产生新的模型版本号（`model_version`）。

- **历史决策卡片永不重算**：每张卡片记录生成时的 `model_version`，保留原始评分
- **得分趋势线标注版本变更**：趋势线上在模型版本变更日期画竖线 + "模型更新"标签，帮助用户理解评分跳变原因
- **新旧卡片可比性**：同一 `model_version` 内的卡片评分可直接比较；跨版本比较需参考版本变更说明

### 3.10 LLM 智能层调用契约

LLM 在分析流程中有三次明确的调用，每次的输入输出结构不同。此契约定义了产品层面的"LLM 应该收到什么、应该输出什么"，是 TRD prompt 工程的产品约束。

**调用一：信息检索与研判（Layer 2，D1/D2/D3 各一次，D4 不调用）**

输入：维度名称和描述、检索策略关键词（§3.3 各维度 LLM 智能层定义）、当前量化基础分和子指标快照、上一次分析的 LLM 研判结果（用于检测变化）。

输出（结构化）：检索到的事件列表（每条含 source_url、source_name、date、summary）；定性研判（direction: bullish/bearish/neutral、magnitude: major/moderate/minor、confidence: high/medium/low、rationale: 一句话理由）。不输出数字调整——数字由 §3.4 映射规则生成。

验证规则（§3.1 LLM 输出验证）：无来源 URL 的事件不生效；magnitude 为 major 需 ≥ 2 个独立信源。

**调用二：综合解读文本生成（Layer 3，每次预计算一次）**

输入：四维最终得分（量化基础 + LLM 调整后）、各维度的 LLM 研判摘要、综合方向和冲突状态、内容风格指南（§13.3）。

输出：market_interpretation（100-200 字综合解读）、risk_factors（2-3 条风险因子，每条 30-50 字）、regime_summary（一句话市场体制判断）。

**调用三：执行计划生成（持仓级，每个持仓标的一次）**

输入：标的最终评分和维度明细、用户持仓信息（标的代码、仓位比例、成本价）、**用户风险偏好（保守/稳健/进取）及对应的参数约束（§7.6）**、同类标的总敞口（用于协调）、支撑/阻力位（richson 计算）、ATR 值（richson 计算）、仓位管理规则（§8.2）。

输出（结构化）：action（总体建议）、default_action（无场景触发时的默认建议）、scenarios（条件分支列表，每条含 condition、action、lot_count、rationale、priority）、stop_loss、take_profit、valid_days、no_trigger_note。

**"今日变化摘要"的基线定义**：评分变化的比较基线统一为**上一次定时更新（T-1）**的评分，不追踪用户最后查看时间。如果当日有事件触发的即时更新，则今日变化 = 最新评分 − 昨日定时更新评分。

## 5. 标的详情页

标的的完整分析报告页。公开可访问（激活标的），执行计划 tab 需登录 + 持仓。

### 5.2 信息架构：Sticky Header + 三 Tab

采用**顶部固定区 + Tab 分页**结构，解决信息密度问题：

- 顶部固定区始终展示核心结论和关键警告，用户无论在哪个 tab 都能看到
- 三个 tab 对应用户的三个自然问题：**为什么？（分析）**、**有什么风险？（风险）**、**该怎么做？（执行）**
- 每个 tab 内容控制在 1-2 屏以内

#### 5.2.1 顶部固定区（Sticky Header，始终可见）

```
┌──────────────────────────────────────────────┐
│  GLD · SPDR Gold Shares                       │
│  $4,750.00  +1.2%                             │
│                                               │
│  中期展望  72/100  中度看涨  · 近一年中高        │
│                                               │
│  今日: D3+5(央行购金报道) D2+3(DXY走弱) → +7  │ ← 变化≥5分时展示
│  ⚠ 宏观利率(30)与结构性需求(85)方向冲突        │ ← 强冲突时展示
│  📍 分析更新于 8 小时前 · 金价已变动 +1.8%      │ ← 偏移>2%时展示
└──────────────────────────────────────────────┘
```

固定区包含：
- 标的名称、当前价格、涨跌幅
- 综合评分 + 方向标签 + 时间维度 + 历史分位
- 今日变化摘要（条件展示：评分变化 ≥ 5 分时显示维度级归因）
- 维度冲突警告（条件展示：强冲突时红色，弱冲突时黄色）
- 数据新鲜度指示器（三级警告）：
  - > ±2%：🟡 黄色提示"金价自上次分析后已变动 X%，当前评分可能未反映最新走势"
  - > ±5%：🟠 橙色警告"金价大幅变动 X%，当前分析可能已失效，请谨慎参考"
  - > ±10%：🔴 红色强警告"金价剧烈波动 X%，当前分析不再可靠，请勿基于此评分做交易决策"（评分数字和方向标签灰度化，视觉上标记为"失效"）

#### 5.2.2 [分析] Tab

回答"为什么是这个评分"：

- **K 线图**：支持 1D/1W/1M/3M/1Y 切换，图上标注 200 日均线和自动计算的支撑/阻力位
- **LLM 综合解读**：100-200 字的市场解读文本（每日预计算生成）
- **四维分析折叠面板 ×4**：每个面板标题显示维度名称 + 得分 + 方向 + 权重%。LLM 有调整时显示两层分数："结构性需求 82 分（量化基础 70 → LLM +12）"。展开后显示全部子指标和 LLM 调整原因。维度名称旁带 **"?"** 图标，点击展示 2-3 句概念解释（如"TIPS 收益率是扣除通胀后的真实利率。当它下降时，持有黄金的机会成本降低，利好金价"）——帮助用户渐进式建立投资认知
- **得分趋势线**：默认 90 天，支持切换 30/90/180/240 天。模型版本变更处标注竖线

#### 5.2.3 [风险] Tab

回答"有什么可能出错"：

- **关键风险因子**：2-3 条"什么会让当前判断失效"，LLM 生成
- **关键价格水平**（自动计算）：
  - 支撑位：20 日唐奇安通道下轨、近 60 日显著低点（反弹 > 3%）、200 日均线——取最接近当前价者
  - 阻力位：20 日唐奇安通道上轨、近 60 日显著高点、历史新高——取最接近当前价者
  - 展示当前价相对支撑/阻力的距离百分比
- **历史回撤参考**：当前牛市中的最大回撤幅度及历史对比
- **事件日历**：未来 7 天与当前标的相关的宏观事件 + Polymarket 概率

#### 5.2.4 [执行] Tab

回答"我该怎么做"：

**未登录或无持仓**：展示**示范执行计划**（Demo Plan）而非完全遮罩，让用户在免费层体验产品最核心的价值：

```

│  建议：逢回调加仓                                │
│  场景一：IF 金价回调至 $4,600 → 加仓 2%          │
│  场景二：IF 金价突破 $5,000 → 加仓 2%            │
│  止损：$4,400                                    │
│──────────────────────────────────────────────────│
│  ⚠ 以上基于示范持仓，非为您定制                   │
│  📌 录入您的真实持仓，获取专属执行计划 →           │
└──────────────────────────────────────────────────┘
```

示范计划由 LLM 调用三动态生成（输入为预设的假设持仓参数：持仓 10%、成本为当前价 ×0.95、稳健型风险偏好），每日随标的级分析一起预计算。Demo Plan 是产品最重要的转化触点（aha moment），值得每日一次 LLM 调用的成本。

**有持仓**：完整条件分支执行计划（见 §8），包含：
- 当前持仓概况（成本、仓位、浮盈亏）

## 7. 持仓管理

### 7.1 录入方式

纯手动录入（MVP 不支持导入或券商 API 对接），支持三种模式，由轻到重渐进式引导：

- **标记模式（最轻量）**：用户只需选择标的 + 大概仓位档位（轻仓 < 10% / 中仓 10-25% / 重仓 > 25%），不需要输入精确成本。系统用当前价作为估算成本，生成"粗略版"执行计划（标注"基于估算，录入精确成本可获得更准确的建议"）
- **快速模式**：填入当前均价成本和仓位比例，获得精确执行计划
- **明细模式**：录入历史买卖交易记录，系统自动计算成本和仓位

三种模式可随时升级：标记模式 → 补充成本和比例升级为快速模式 → 补充交易记录升级为明细模式。这把"录入持仓"从一次性高门槛动作变成渐进式过程。

### 7.2 标的选择交互

先按类型分类浏览，再在分类内搜索选择。MVP 阶段仅黄金类标的可选（其他类型置灰不可选）。

### 7.3 录入字段

**快速模式**：标的名称（从标的库选择）、当前均价成本、当前仓位比例（占总资金 %）

**明细模式**：时间、价格、数量、方向（买入/卖出）

### 7.4 自动计算

- 综合持仓成本（加权平均）
- 当前浮盈亏
- 历史操作记录

### 7.5 限制

MVP 每用户最多 5 个持仓标的。

### 7.6 风险偏好设置

用户首次录入持仓时（或在设置中随时调整），选择投资风格偏好：

- **保守型**：更看重资产安全，能接受较低收益
- **稳健型**：希望平衡收益和风险（默认选项）
- **进取型**：能承受较大波动，追求更高收益

风险偏好影响执行计划的风险参数：

| 参数 | 保守型 | 稳健型 | 进取型 |
|---|---|---|---|
| 单笔加仓上限 | 2% | 5% | 8% |
| 止损宽度 | 1.5×ATR | 2×ATR | 3×ATR |
| 集中度蓝色提示线 | 10% | 15% | 20% |
| 评分触发加仓阈值 | ≥ 70 | ≥ 60 | ≥ 55 |

风险偏好不影响标的级分析评分（评分对所有用户相同），只影响持仓层的执行计划生成参数。

## 8. 执行计划与仓位管理

### 8.1 执行计划输出结构

采用**条件分支**结构（IF/THEN），非线性步骤：

```
执行计划 = {
  action: "逢回调加仓",
  default_action: "维持现有仓位，等待明确信号",  // 无场景触发时的默认建议
  current_position: 8%,
  target_position: 12%,
  scenarios: [
    { condition: "金价回调至支撑位", action: "加仓 2%", rationale: "支撑确认" },
    { condition: "金价突破阻力位", action: "加仓 2%", rationale: "趋势延续" },
    { condition: "金价跌破关键支撑", action: "减仓至 4%", rationale: "止损保护" }
  ],
  stop_loss: 基于 2×ATR(14),
  take_profit: 基于阻力位+动量,
  valid_days: 7,
  no_trigger_note: "若 7 天内无场景触发，维持现有仓位，下期自动刷新"
}
```

**默认建议**：每个执行计划除条件分支外，包含一个 `default_action`——当所有场景均未触发时的明确建议（通常是"维持现有仓位"）。"不动"本身就是一个专业判断，用户需要知道等待也是策略的一部分。

**时间维度**：综合评分 = 中期展望（1-3 月），执行计划 = 短期有效（7 天），到期后由下一次预计算自动刷新。UI 上分别标注。

**场景交互规则**：条件分支场景之间非独立执行，遵循以下规则：
- **止损优先**：止损/减仓场景（如"跌破关键支撑"）的优先级永远高于加仓场景。系统在生成场景时自动标注优先级
- **加仓后止损上移**：任一加仓场景执行后，止损位自动调整为新加仓的成本价（保本止损），而非继续使用原始止损位
- **场景互斥**：同一方向的场景不可叠加执行（如不能在支撑位加仓 2% 后又在阻力位加仓 2%，除非第一次加仓后执行计划已刷新）

### 8.2 仓位管理原则

- **分数 Kelly 准则**：单笔不超过半 Kelly，零售投资者不超过 5%
- **ATR 止损**：止损位基于 2×ATR(14)
- **金字塔建仓**：分 2-3 批次，每批独立触发条件
- **加仓基于模型信号，非盈亏状态**：综合评分 < 40 时不建议加仓。评分 ≥ 60 且无强冲突时，即使浮亏仍可建议在支撑位加仓，但附加风险提示
- **集中度控制**：见 §8.3
- **同类标的执行协调**：当同一二级分类下存在多个持仓（如 GLD + 518880 都是黄金），加仓建议在标的间协调分配，避免独立执行后总敞口超标。LLM 调用三的输入包含同类总敞口信息，由 LLM 在生成执行计划时统筹分配

### 8.3 持仓集中度分级检查

同一二级分类下合计仓位超阈值时，在标的详情页执行计划区、投研简报、持仓管理页面展示：

| 合计仓位 | 级别 | 展示 |
|---|---|---|
| ≥ 15% | 蓝色提示 | "黄金配置已达 X%，处于机构建议区间上限" |
| ≥ 25% | 橙色警告 | "黄金配置集中度较高，请注意分散风险" |
| ≥ 35% | 红色强警告 | "黄金配置严重集中，强烈建议控制敞口" |

## 10. 通知推送

### 10.1 推送时间

所有时间以北京时间（UTC+8）为准：

| 推送 | 时间 | 覆盖标的 |
|---|---|---|

### 10.2 推送渠道

MVP：**邮件（HTML 邮件）**——实现成本低，覆盖面广，无需第三方平台审核。

后续：微信公众号（模板消息）、飞书机器人、钉钉、Slack、Telegram、iOS APNs。采用可插拔适配器模式，新增渠道只需添加一个 adapter。

### 10.3 推送内容分类

| 类型 | 触发条件 | 接收人 | 内容 |
|---|---|---|---|
| **每日简报** | 每日早 8:30 定时 | 所有已注册用户 | 市场体制 + 黄金评分（较昨日变化）+ 今日关注事件 + 持仓建议摘要（如有持仓） |
| **每周投研洞察** | 每周一早 8:30 | 所有已注册用户 | 上周回顾 + 本周展望 + 一个投资知识点（300-500 字，LLM 生成） |
| **市场快讯** | 事件驱动即时更新，评分变化 ≥ 10 分 | 所有已注册用户 | "黄金评分从 68 升至 80，主要驱动：[变化摘要]" |
| **持仓建议** | 日常更新后执行计划变化 | 对应持仓用户 | 个性化执行计划摘要 |

**每日简报**是产品的日常习惯锚点——即使什么都没变，用户也收到一封确认"今天不需要操作"的邮件。确认"不需要行动"本身就是有价值的信息。内容从预计算结果中提取，无需额外 LLM 调用。

**每周投研洞察**解决低波动率期间的用户参与度下降问题。在市场平静的周里，每日简报内容几乎不变，用户容易失去兴趣。周报提供更深度的内容——回顾上周宏观动态、展望本周关键事件、讲解一个投资概念（如"什么是金银比"、"央行为什么买黄金"）。每周一次 LLM 调用，成本可忽略，但显著提升低波动期的用户粘性和教育价值。

### 10.4 架构

可插拔适配器模式。统一接口 `send(userId, message, channels[])`，每个渠道一个 adapter。MVP 仅实现 `EmailAdapter`，新增渠道只需添加对应 adapter 文件，零改动核心逻辑。

## 12. 数据源

### 12.1 数据源总览

所有金融数据源由 richson（Python 量化服务）统一管理：

| 数据源 | 用途 | 更新频率 | 服务归属 |
|---|---|---|---|
| FRED (`fredapi`) | 宏观：TIPS、DXY、Fed BS、M2、TGA、利率、通胀、信用利差 | 日级 | richson |
| Yahoo Finance (`yfinance`) | 黄金价格（GC=F）、ETF 价格、ETF 持仓量 | 日级 | richson |
| AKShare | A 股 ETF 价格和估值 | 日级 | richson |
| Polymarket API | 宏观事件隐含概率 | 日级 + 事件触发 | richson |
| CFTC COT | 期货持仓数据 | 周度 | richson |
| LLM（Claude/OpenAI） | 解读文本、信息检索、分数调整 | 每次预计算时 | richman |

### 12.2 LLM 配置

多 provider 抽象层，支持用户自带 key（BYOK）和系统默认 key。降级策略：用户 key → 系统默认 key → 模板生成。标的级分析使用平台配额，持仓级分析使用用户 key 或系统默认。

### 12.3 Polymarket 集成

- 美联储利率决议概率 → D1 子信号（5%）
- 地缘政治事件概率 → D3 子信号（10%）
- 黄金价格合约 → 事件雷达展示，不参与评分
- 数据旁标注"来自预测市场，存在流动性和准确性风险"

**监管依赖风险与替代方案**：Polymarket 面临 CFTC 监管调查和部分地区访问限制。系统架构不应对其产生硬依赖：
- **D1 替代**：CME FedWatch（美联储利率预期）可替代降息概率信号，公开免费
- **D3 替代**：地缘事件概率可由 LLM 智能层的联网搜索补充（已有此能力）
- **事件雷达替代**：Kalshi（CFTC 批准的美国预测市场）或公开经济日历 API
- Polymarket 不可用时，按 §3.7 降级策略处理（权重归零，信心度扣 5%），同时自动切换至替代数据源

### 13.1 免责声明内容

"百万·Richman 提供的分析和建议仅供参考，不构成投资建议或投资要约。投资有风险，入市需谨慎。用户应基于自身财务状况和风险承受能力独立做出投资决策。"

### 13.2 展示位置

| 位置 | 方式 |
|---|---|
| 标的详情页底部 | 固定灰色小字（不可关闭） |
| 执行计划区顶部 | 一行简短提示 |
| 注册流程 | 勾选"已阅读并理解免责声明" |
| 首次查看执行计划 | 弹窗确认（仅一次） |

所有决策卡片避免使用"保证"、"确定"、"必须"等绝对性措辞。

### 13.3 内容风格指南

所有 LLM 生成的面向用户文本（综合解读、维度解读、风险因子、市场体制判断、今日变化摘要）遵循统一的产品声音：

- **语气**：专业、克制、有主见。像资深基金经理在晨会上做简报——有明确方向判断，但承认不确定性
- **结构**：先结论后原因。"当前看涨（结论），主要因为...（原因）"
- **数据引用**：引用具体数据时标注来源和日期，如"据路透社 4/10 报道..."
- **禁止用语**：不使用"一定"、"必然"、"稳赚"、"暴涨"、"崩盘"等绝对性或情绪化词汇
- **长度约束**：综合解读 100-200 字，维度解读 50-100 字，风险因子每条 30-50 字
- **语言**：跟随用户语言设置（中文/英文），LLM prompt 中明确指定输出语言

### 13.4 数据隐私

- 用户持仓数据（标的、成本、仓位比例）不发送给任何第三方。标的级分析由平台预计算，不需要用户持仓信息；持仓级分析的 LLM 调用仅发送标的代码和评分，不发送用户成本和仓位
- 用户删除账户时，持仓数据在 30 天内彻底删除
- 不基于单个用户的持仓数据进行商业化或聚合分析
- 用户的 LLM API Key（BYOK）加密存储，不明文记录在日志中

### 15.3 后端：richman（Go）+ richson（Python）

**richman（Go 主服务）**：

| 项 | 选型 |
|---|---|
| 语言 | Go (Gin + sqlc) |
| 数据库 | PostgreSQL |
| LLM | 多 provider 抽象层（Claude / OpenAI），Agent 框架采用 Google ADK |
| 定时任务 | Cron 调度器 |
| 部署 | Docker + VPS |

职责：用户认证、持仓 CRUD、定时调度、LLM 编排（信息检索 + 分数调整 + 文本生成）、决策卡片生成与持久化、执行计划生成、通知推送、API 服务。**不直接调用任何金融数据 API**。

**richson（Python 量化服务）**：

| 项 | 选型 |
|---|---|
| 语言 | Python |
| 数据源 | FRED (fredapi)、Yahoo Finance (yfinance)、AKShare、Polymarket API、CFTC COT |
| 量化工具 | pandas-ta-classic、statsmodels、pyarrow |
| 部署 | Docker（与 richman 同一 docker-compose） |

职责：全部金融数据获取（FRED、Yahoo、AKShare、Polymarket、COT）、数据处理（频率对齐、百分位排名、净流动性计算）、技术指标计算、维度评分计算、冲突检测、支撑/阻力位计算。**不涉及用户、LLM、持仓、通知等业务逻辑**。

**协作流程**：richman 调用 richson 获取量化基础分 → richman 调用 LLM 执行信息检索和分数调整 → richman 合成最终评分并持久化。具体通信协议在 TRD 中定义。

后端三层架构：API handlers → Service → Repo。

### 15.4 LLM 在产品中的双重角色

**角色一：信息检索与分析引擎**（Layer 2）。LLM 利用联网搜索能力，按产品预定义的检索策略（§3.3 各维度 LLM 智能层），主动获取实时新闻、官员讲话、机构报告等非结构化信息，分析提炼后输出结构化的维度分数调整建议（±15 上限）+ 调整理由。由 richman（Go）编排调用。

**角色二：文本生成引擎**（Layer 3）。基于量化底座 + LLM 调整后的最终评分，生成综合解读文本、风险因子描述、事件日历注释、市场体制一句话判断等面向用户的自然语言内容。由 richman（Go）编排调用。

### 15.5 API 设计原则

- 纯 RESTful JSON，camelCase 字段
- URL 路径 kebab-case 复数形式
- 分页：page + pageSize
- 统一错误格式：`{ error: { code, message, details } }`
- Market Overview 相关 API 无需认证
- Portfolio / Briefing 相关 API 需 JWT 认证
