# TRD 审查纪律

本文档定义 TRD 编写和审查过程中必须遵循的纪律，防止系统性遗漏。

## 1. 代码库先行原则

写 TRD 中任何涉及现有代码的设计之前，必须先读代码确认现有状态：

| 写什么 | 必须先做什么 |
|--------|------------|
| ALTER TABLE ADD COLUMN | grep 现有 migration 确认列不存在 |
| ALTER TABLE 改约束 | 读当前 migration 确认约束名和允许值 |
| 新 Repo struct 的字段类型 | 读现有 repo 文件确认实际使用的类型（如 pgxpool.Pool vs sql.DB）|
| migration 脚本中的事务控制 | 读 runner.go 确认是否自动包裹事务 |
| 新 API 端点路径 | grep 现有路由注册确认无冲突 |
| 前端 HTTP client 设计 | 读现有 client.ts 确认当前封装模式 |
| 前端组件 API 用法 | grep node_modules 中对应 .d.ts 确认参数名 |

违反此原则的借口不存在："我记得是这样的"不是证据，代码才是。

## 2. 第三方 API 验证原则

写任何第三方库的 API 名称（import path、方法名、参数名、组件 prop）之前，必须从以下来源之一确认：

- npm/PyPI/Go modules 的当前版本文档
- 项目实际安装的源码或 .d.ts 文件
- Context7 MCP 工具或 web search

不允许凭训练记忆直接写入。覆盖范围：

- Python: 包名、import path、类名、方法签名、参数名（如 `instruction` vs `instructions`）
- Go: 模块路径、struct 名、方法签名
- 前端: 组件 prop 名、CSS 变量名、配置 key、hook 返回值结构
- 框架工具: CLI 参数、配置文件格式

## 3. 端到端数据链路追踪

每个核心数据点必须有完整的链路追踪，跨越所有相关 TRD：

```
[数据产生] -> [存储位置] -> [API 传输] -> [前端消费]
 richson 计算   DB 表.列     richman 响应字段   组件 + hook
```

TRD 编写完成后，对以下数据点逐条验证链路完整性：

- 综合评分 + 各维度分数
- 执行计划（action、scenarios、stop_loss 等）
- 回撤参考数据
- 汇率转换
- 变化摘要
- 冲突检测结果
- 事件日历
- 模型版本

链路中任何一环缺失（如 richson 计算了但 DB 没列、DB 有列但 API 没返回、API 返回了但前端没消费），视为 CRITICAL 问题。

## 4. 多角色审查矩阵

自查不允许用同一视角重复多轮。必须按以下角色各审查一轮，每个角色有固定的 checklist：

| 角色 | 必查项 |
|------|--------|
| DBA | DDL 语法、索引覆盖、事务边界、锁影响、类型精度、约束冲突 |
| 后端研发 | 方法签名与现有代码兼容、事务范围、并发安全、错误码覆盖 |
| 前端研发 | 组件 prop 与 API 响应字段对齐、依赖版本、barrel 导出、i18n 覆盖 |
| 安全 | 认证边界、敏感数据传输/存储、注入面、限流、CORS |
| SRE | 健康检查、优雅关闭、日志脱敏、备份、容量规划 |
| QA | 竞态条件、边界值、异常路径用户可见行为、状态机完整性 |
| 产品 | PRD 逐节功能覆盖、用户旅程完整性、MVP 边界清晰度 |

## 5. 已知问题必须落地

识别问题后只有两种合法状态：

1. **当场修复** -- 直接改 TRD
2. **写入"已知问题"章节** -- 每份 TRD 必须有"已知问题与编码阶段必须处理项"章节，每项包含：问题描述 + 具体处理方案

不合法的状态：
- "记在审查报告里" -- 审查报告不是持久产出物
- "编码阶段会处理" -- 没写进 TRD 就等于不存在
- "这是 PRD 的问题" -- TRD 必须记录偏差决策和待 PRD 修正的项
- "这应该是一次性问题" -- 规则存在的意义就是拦截"应该不会再发生"的问题

## 6. 跨 TRD 一致性检查清单

多份 TRD 并存时，完成后必须检查：

- [ ] 同一个 DB 列在所有 TRD 中名称和类型一致
- [ ] richson API 端点在 richman TRD 中都有代理/调用设计
- [ ] richman API 响应字段在 frontend TRD 中都有组件消费
- [ ] 枚举值（风险偏好、信号级别等）在所有 TRD 中统一
- [ ] 迁移编号无冲突、执行顺序有记录
- [ ] 章节交叉引用目标存在且内容匹配
- [ ] 错误码覆盖所有错误场景
